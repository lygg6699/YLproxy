## 🔍 第 1 步：需求分析

### 任务：理解清楚再开始

**时间**：5-10 分钟

**检查清单**：

```
需求分析检查表：

□ 任务标题清晰
  - 示例：✅ "实现数据导入功能"
  - 反例：❌ "改进性能"

□ 任务范围明确
  - 包括什么？（具体功能点）
  - 不包括什么？（确认边界）

□ 成功标准清晰
  - 功能应该如何工作？
  - 如何验证成功？
  - 是否有性能指标？

□ 依赖关系清晰
  - 是否依赖其他任务？
  - 是否阻塞其他任务？

□ 风险评估
  - 技术风险（低/中/高）
  - 影响范围（小/中/大）
```

**示例对话**：

```
任务：优化 ProxyTester 的性能

Q: 期望的优化目标是什么？
A: 从当前 5 秒减少到 3 秒

Q: 涉及哪些模块？
A: 只修改 ProxyTester.cs，不改其他模块

Q: 如何验证优化成功？
A: 运行测试，测试通过且耗时 < 3 秒

Q: 是否会影响其他功能？
A: 不会，只是内部优化

分析：低风险，范围明确 ✅
```

## 📊 第 2 步：影响分析

### 任务：预测修改的影响

**时间**：5-15 分钟

**分析维度**：

#### A. 代码依赖关系

```
修改 ProxyTester.cs
    │
    ├─ 直接依赖者：
    │  ├─ MainViewModel（调用 TestAsync）
    │  └─ ProxyService（调用 TestAsync）
    │
    ├─ 间接依赖者：
    │  ├─ MainWindow（通过 MainViewModel）
    │  └─ AddProxyWindow（通过 ProxyService）
    │
    └─ 影响范围：
       └─ UI 层、应用层可能受影响
```

#### B. 配置影响

```
是否需要修改配置文件？

□ AppSettings.json
  - 新增配置项？
  - 删除配置项？
  - 修改配置项？

□ data/config.json
  - 新增字段？
  - 格式变更？
  - 需要迁移脚本？
```

#### C. 数据影响

```
是否影响数据结构？

□ 模型类（Models/）
  - ProxyItem 需要修改？
  - 需要新增类？

□ 序列化格式
  - JSON 结构变更？
  - 需要向后兼容？

□ 数据库
  - 表结构变更？
  - 需要迁移脚本？
```

#### D. 日志影响

```
需要添加新日志？

□ 调试日志
  - 关键步骤记录？

□ 错误日志
  - 异常情况记录？

□ 性能日志
  - 耗时统计？
```

#### E. 文档影响

```
哪些文档需要更新？

□ 代码注释
  - 方法文档？
  - 参数说明？

□ 项目文档
  - README.md？
  - 功能说明？

□ 架构文档
  - 设计变更？
  - 新增模块？
```

**输出**：影响分析报告

```
修改 ProxyTester 的性能优化

直接影响文件：
- ProxyTester.cs

间接影响文件：
- MainViewModel.cs (调用 TestAsync)
- ProxyService.cs (调用 TestAsync)

配置影响：
- AppSettings.json: 新增 TestTimeout 配置

数据影响：
- 无结构变更

日志影响：
- 新增 DEBUG 日志：优化前后耗时对比

文档影响：
- 更新 03-架构设计/后端架构.md

总体评估：低风险，范围明确 ✅
```

## ✏️ 第 3 步：代码修改

### 任务：按规范修改代码

**时间**：10-30 分钟（取决于复杂度）

**编码规范**：参考 [coding-rules.md](coding-rules.md)

**关键检查**：

```csharp
// 1. 命名规范
public class ProxyTester  // ✅ PascalCase 类名
{
    private readonly ILogger _logger;  // ✅ camelCase 私有字段
    
    public async Task<ProxyTestResult> TestAsync(ProxyItem proxy)  // ✅ 异步方法
    {
        // 2. 日志完整性
        _logger.Debug($"处理整数: {number}");
        
        try
        {
            // 3. 异常处理
            var result = await PerformTestAsync(proxy);
            
            _logger.Info($"测试完成: {proxy.Name}, 结果: {result.Success}");
            return result;
        }
        catch (Exception ex)
        {
            _logger.Error($"测试异常: {ex.Message}");
            throw;  // ✅ 记录后重新抛出
        }
    }
}
```

**提交规范**：

```
修改文件清单：
- ProxyTester.cs (优化性能)
- MainViewModel.cs (更新测试超时时间)

新增日志点：
- DEBUG: 性能优化前后耗时
- INFO: 测试完成时间统计

配置变更：
- AppSettings.json: 新增 TestTimeout 配置

文档更新：
- 已添加方法注释
- 将同步更新 README
```

## 🔨 第 4 步：编译验证

### 任务：确保编译成功

**时间**：2-5 分钟

```powershell
# 清理旧编译
dotnet clean

# 编译项目
dotnet build --configuration Debug

# 检查清单
□ Exit Code = 0（成功）
□ 无编译错误
□ 无新增编译警告
```

**常见问题**：

| 错误 | 解决方案 |
|---|---|
| `CS1002: ; 预期` | 检查语法（通常是括号不匹配） |
| `CS0246: 类型或命名空间找不到` | 检查 using 语句或项目引用 |
| `CS0019: 运算符不能应用于...` | 检查类型转换 |

## 🧪 第 5 步：单元测试

### 任务：运行测试并修复失败

**时间**：5-10 分钟

```powershell
# 运行所有测试
dotnet test --no-build

# 运行特定测试
dotnet test --filter "ClassName"

# 检查清单
□ 所有测试通过
□ 无新增失败
□ 代码覆盖率正常
```

**失败处理**：

```
如果测试失败，分析失败原因：

1. 修改了 API，但测试未更新
   → 更新测试代码

2. 引入了 bug
   → 修复代码逻辑

3. 环境问题（如端口占用）
   → 清理环境后重试
```

## 🔬 第 6 步：集成测试

### 任务：验证功能端到端工作

**时间**：10-20 分钟

**Smoke Test 检查表**：

参考 [test-rules.md](test-rules.md)

```
测试项            期望结果
─────────────────────────
启动应用          ✅ 正常启动
加载配置          ✅ 无错误
功能 X            ✅ 符合预期
监听端口          ✅ 端口打开
日志输出          ✅ 有适当日志
停止应用          ✅ 正常退出
```

**手动验证步骤**：

```powershell
# 1. 启动应用
.\bin\Debug\net10.0\MyApp.exe

# 2. 在 UI 中执行功能
# - 添加记录
# - 行业业务
# - 启动服务
# - 检查日志

# 3. 查看日志
Get-Content logs/app.log | Select-String "功能名（示例）"

# 4. 检查是否有新错误
Get-Content logs/app.log | Select-String "ERROR"
```

## 📝 第 7 步：文档更新

### 任务：同步所有文档

**时间**：5-10 分钟

**检查清单**：

```

□ 功能文档已更新
```
