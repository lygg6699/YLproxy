# Development Guard v1.0

> 开发守护协议：确保代码质量、功能完整性、文档同步

版本：V1.0  
生效日期：2026-07-15  
维护方：开发团队  
适用对象：开发者、AI 代理、自动化流程

---

## 📋 核心原则

### 基本信条

任何代码修改都必须遵循以下四层防线：

```
代码修改
    ↓
第 1 层：编译验证（Build Check）
    ↓
第 2 层：单元测试（Unit Tests）
    ↓
第 3 层：功能测试（Integration Tests）
    ↓
第 4 层：文档同步（Documentation）
    
所有层级必须通过才能标记为完成
```

---

## 🎯 修改前的分析义务

在任何代码修改前，**必须执行**：

### 1. 需求分析

- [ ] 理解任务的具体目标
- [ ] 确认修改范围（影响哪些模块）
- [ ] 评估风险等级
- [ ] 列出需要修改的文件

### 2. 影响范围分析

**必须验证**以下内容：

```
修改的代码
    ↓
├─ 直接依赖：哪些类/接口依赖此模块
├─ 间接依赖：哪些模块的模块依赖此模块
├─ 数据流：是否影响数据序列化/反序列化
├─ 配置：是否需要更改配置文件
├─ 日志：是否需要添加新日志点
└─ 文档：哪些文档需要同步更新
```

### 3. 审视类似代码

- [ ] 检查是否已有类似实现（避免重复）
- [ ] 检查是否有可复用的工具函数
- [ ] 检查是否有相同的模式（如异常处理、日志方式）

## 🔧 修改时的编码义务

### 严格遵循的规则

#### 规则 1：编码规范

参考：[.guard/coding-rules.md](coding-rules.md)

**关键检查点**：

```csharp
// ❌ 禁止：硬编码路径
string path = "C:\\data\\config.json";

// ✅ 推荐：从配置或解析器读取
string path = _pathResolver.GetConfigPath();

// ❌ 禁止：无日志的异常吞没
try { ... } catch { }

// ✅ 推荐：记录日志
try { ... } catch (Exception ex) { 
    Logger.Error($"操作失败: {ex.Message}"); 
    throw; 
}

// ✅ 推荐：新功能有日志输出
Logger.Info($"功能 X 已启动，参数：{param}");
```

#### 规则 2：不允许修改无关代码

- [ ] 不修改其他模块的代码（除非明确要求）
- [ ] 不修改配置文件（除非任务明确需要）
- [ ] 不删除已有功能（除非明确弃用）

#### 规则 3：异常处理完整性

```csharp
// 所有异常都必须：
// 1. 记录日志（ERROR 级别）
// 2. 可选：转换为业务异常
// 3. 要么处理，要么重新抛出

public void ImportantOperation()
{
    try
    {
        // 业务逻辑
    }
    catch (OperationCanceledException ex)
    {
        Logger.Warning($"操作被取消: {ex.Message}");
        throw;  // 重新抛出
    }
    catch (Exception ex)
    {
        Logger.Error($"关键错误: {ex.GetType().Name} - {ex.Message}");
        throw new ApplicationException("操作失败", ex);
    }
}
```

## ✅ 修改后的验证义务

### 四层验证必须全部通过

#### 验证 1：编译（Build）

```powershell
# 必须成功
dotnet build --configuration Debug

# 检查清单
□ 编译成功（Exit Code = 0）
□ 无编译错误
□ 无新增编译警告（必须修复或抑制）
```

#### 验证 2：单元测试（Unit Tests）

```powershell
# 必须通过
dotnet test --no-build

# 检查清单
□ 所有测试通过
□ 无新增失败
□ 代码覆盖率不下降（如有目标）
```

#### 验证 3：集成测试（Smoke Test）

参考：[.guard/test-rules.md](test-rules.md)

```
检查项          期望结果
────────────────────────
启动应用        ✅ 正常启动
加载配置        ✅ 无错误
创建代理        ✅ 能添加
启动代理        ✅ 能启动
监听端口        ✅ 端口打开
停止代理        ✅ 能停止
退出应用        ✅ 正常退出
```

#### 验证 4：文档同步（Documentation）

```
检查项              状态
──────────────────────────
修改代码               ✅
更新对应文档           ⏳ 必须
更新 README 导航       ⏳ 如需要
更新 CHANGELOG         ⏳ 如需要
```

## 📋 任务完成的验收标准

### 任务必须满足以下条件才能标记为"完成"

| 条件 | 检查内容 | 状态 |
|---|---|---|
| **1. 编译通过** | `dotnet build` 成功，无错误 | ✅ 必须 |
| **2. 测试通过** | 所有单元测试通过 | ✅ 必须 |
| **3. 功能验证** | Smoke Test 全部通过 | ✅ 必须 |
| **4. 代码审查** | 遵循编码规范 | ✅ 必须 |
| **5. 日志完整** | 新功能有适当日志 | ✅ 必须 |
| **6. 文档同步** | 相关文档已更新 | ✅ 必须 |
| **7. 无回归** | 未破坏已有功能 | ✅ 必须 |

### 检查表（任务完成前必须全部勾选）

```
□ 编译成功（无错误、无新增警告）
□ 单元测试全部通过
□ Smoke Test 全部通过
□ 修改文件清单已列出
□ 编码规范已检查
□ 日志输出已验证
□ 文档已更新
□ 无回归问题发现
□ 生成报告已审查
```

## ❌ 禁止事项

### 严格禁止的操作

1. **❌ 编译失败就标记完成**
   - 必须修复所有编译错误
   - 必须修复所有编译警告

2. **❌ 跳过测试**
   - 所有测试必须运行
   - 任何测试失败必须排查

3. **❌ 不更新文档**
   - 代码改了，文档必须改
   - API 改了，使用说明必须改

4. **❌ 硬编码敏感信息**
   - 禁止硬编码密码、端口、IP
   - 必须从配置读取

5. **❌ 沉默的失败**
   - 所有异常必须记录
   - 不允许空的 catch 块

6. **❌ 修改无关代码**
   - 只修改必要的文件
   - 不要 cleanup 其他模块

## 🔄 流程图

```
收到任务
    │
    ▼
[分析需求 & 影响范围]
    │
    ├─ 需求清晰？
    │  ├─ 否 → 澄清需求
    │  └─ 是 ↓
    │
    ├─ 影响范围清晰？
    │  ├─ 否 → 分析影响
    │  └─ 是 ↓
    │
    ▼
[修改代码]
    │
    ├─ 遵循编码规范？
    │  ├─ 否 → 调整代码
    │  └─ 是 ↓
    │
    ├─ 异常处理完整？
    │  ├─ 否 → 补充日志
    │  └─ 是 ↓
    │
    ▼
[执行验证脚本]
    │
    ├─ 编译成功？
    │  ├─ 否 → 修复错误 → 回到编译
    │  └─ 是 ↓
    │
    ├─ 测试通过？
    │  ├─ 否 → 修复测试 → 回到测试
    │  └─ 是 ↓
    │
    ├─ Smoke Test 通过？
    │  ├─ 否 → 修复功能 → 回到修改
    │  └─ 是 ↓
    │
    ▼
[更新文档]
    │
    ├─ 代码文档已更新？
    │  ├─ 否 → 更新文档
    │  └─ 是 ↓
    │
    ├─ README 已更新？
    │  ├─ 否 → 更新导航
    │  └─ 是 ↓
    │
    ▼
[生成报告]
    │
    ▼
[标记完成]
    │
    ▼
任务结束
```
