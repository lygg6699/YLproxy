# .guard/ - 通用开发守护协议

> 可复用到任何 .NET 项目的开发方法论和规范框架

**版本**：V1.0  
**维护方**：开发团队  
**可复用性**：✅ 100% 通用（不涉及项目特定路径或示例）  

---

## 📌 目的

`.guard/` 目录包含一套通用的开发方法论和规范，设计用于：

- 提供统一的代码质量标准
- 规范化开发流程和测试步骤
- 建立清晰的自检机制
- 标准化任务报告格式

**这些文件不包含任何项目特定的内容**，可以直接复用到其他 .NET 项目中。

---

## 📁 文件清单

### 1. **guard.md** - 开发守护协议（核心原则）

**用途**：定义开发工作的基本约束和验证框架  
**内容**：
- 四层验证系统（基础约束、修改义务、验证要求、禁止清单）
- 开发期间的行为规范
- 错误处理和异常处理原则

**适用场景**：
- 所有开发任务的约束框架
- 代码审查时的参考标准
- 项目治理时的基础原则

---

### 2. **workflow.md** - 10 步开发流程

**用途**：提供标准化的任务执行步骤  
**内容**：
- 10 个顺序步骤（需求分析 → 影响评估 → 方案制定 → 执行 → 验证 → 文档 → 报告）
- 每个步骤的检查清单
- 常见错误的诊断建议

**适用场景**：
- 任何新功能开发
- Bug 修复
- 重构任务
- 文档更新

**执行顺序**：必须按 10 步顺序执行，不可跳步或乱序

---

### 3. **coding-rules.md** - C# 编码规范（通用）

**用途**：统一的代码质量标准  
**内容**：
- 命名规范（PascalCase / camelCase / UPPER_SNAKE_CASE）
- 代码组织结构（文件、类、成员顺序）
- 异常处理标准
- 日志规范
- 配置管理模式
- 依赖注入最佳实践

**适用场景**：
- 代码审查时的参考
- 新手开发者的学习材料
- 自检代码质量
- IDE 配置的参考

**注意**：文档中的示例是通用的（`MyProject`, `DataService` 等），项目需要将这些映射到实际类名。

---

### 4. **test-rules.md** - 测试规范（框架）

**用途**：统一的测试标准和 Smoke Test 框架  
**内容**：
- Smoke Test 执行规则和框架
- 单元测试标准
- 集成测试场景
- 性能基准测试
- 回归测试过程

**适用场景**：
- 制定项目特定的测试计划
- 验证代码质量
- 性能监控

**项目集成**：
项目需要在 `docs/README.md` 中定义具体的 Smoke Test 检查清单。  
示例见 YLproxy 的 `docs/development-deployment-outline/README.md`。

---

### 5. **review-rules.md** - 自检规范（通用检查清单）

**用途**：任务完成前的自检框架  
**内容**：
- 8 组检查清单（编译 → 测试 → 功能 → 代码质量 → 文档 → 范围 → 风险 → 最终审查）
- 每项的详细检查点
- 不通过时的处理指导

**适用场景**：
- 所有任务完成前的自检
- 代码审查的标准
- 质量把控的最后关卡

**执行时机**：任务完成后、报告生成前（必须 100% 通过）

---

### 6. **report-template.md** - 报告模板（标准格式）

**用途**：标准化任务完成报告  
**内容**：
- 完整报告格式（头部 → 基本信息 → 修改清单 → 验证结果 → 代码质量 → 风险分析）
- 快速报告格式（适合简单任务）
- 报告检查清单

**适用场景**：
- 所有任务完成时的输出
- 变更追踪和审计
- 项目历史记录

**使用方式**：
1. 选择报告模板（完整 or 快速）
2. 按照结构填充内容
3. 生成最终报告

## 🔄 与项目的集成方式

### 第 1 层：`.agent` → 指向 `.guard/`

项目的 `.agent` 文件应该：
- 引用 `.guard/` 中的这些规范
- 不重复规范内容
- 在需要时指向具体的 `.guard/` 文件

示例：
```
约束 1：10 步执行流程（必须按顺序）
→ 详细内容见 .guard/workflow.md
```

### 第 2 层：`.guard/` → 提供通用方法论

`.guard/` 文件提供：
- ✅ 通用的步骤和规范
- ✅ 通用的代码示例
- ✅ 通用的检查清单框架
- ❌ 不包含项目特定的路径
- ❌ 不包含项目特定的工具名

### 第 3 层：`docs/` → 项目特定的执行手册

项目需要在 `docs/` 中定义：
- 项目特定的 Smoke Test 检查清单
- 项目特定的构建命令
- 项目特定的测试命令
- 项目特定的部署流程
- 项目特定的文档更新规则

## 📊 通用性验证

✅ **已通用化的内容**：
- 所有标题和描述中移除了项目名称
- 代码示例使用通用类名（MyProject, DataService 等）

- 项目特定的类和方法名
- 项目特定的文件路径
- 项目特定的 Smoke Test 清单
- 项目特定的构建配置

## 🚀 使用流程

### 对于 AI 代理
1. 阅读 `.agent` 了解执行约束
2. 参考 `.guard/workflow.md` 执行 10 步流程
3. 参考 `.guard/coding-rules.md` 检查代码质量
4. 参考 `.guard/test-rules.md` 验证功能
5. 使用 `.guard/report-template.md` 输出报告

### 对于新项目集成
1. 复制 `.guard/` 目录到新项目
2. 在 `.agent` 中配置指向 `.guard/` 的规范
3. 在 `docs/` 中定义项目特定的执行手册
4. 将 `.guard/` 中的通用规范映射到项目实际情况

### 对于代码审查
1. 使用 `.guard/coding-rules.md` 作为审查标准
2. 使用 `.guard/review-rules.md` 的检查清单
3. 基于 `.guard/workflow.md` 确认完成度

## 📝 维护和更新

### 何时更新 `.guard/` 文件

**应该更新**（通用方法论改进）：
- ✅ 发现新的通用最佳实践
- ✅ 改进通用的检查清单框架
- ✅ 更新编程语言版本的通用规范

**不应该更新**（项目特定内容）：
- ❌ 项目特定的工具或配置
- ❌ 项目特定的代码示例
- ❌ 项目特定的流程步骤

### 版本控制

`.guard/` 中的文件应该：
- 保持相对稳定（不频繁变更）
- 进行版本控制（在文件头标记版本）
- 有变更日志（记录更新原因）

## 💡 常见问题

**Q: 我可以修改 `.guard/` 中的规范吗？**  
A: 可以，但应该：
1. 确保修改是通用的（不是项目特定的）
2. 通知其他项目使用者（如果已被复用）
3. 记录修改原因和版本

**Q: 项目特定的规范应该放在哪里？**  
A: 放在 `docs/` 目录中，特别是 `docs/README.md`。

**Q: 如何在新项目中使用 `.guard/` ？**  
A: 复制整个 `.guard/` 目录，然后在 `.agent` 中配置引用。
