## 📋 测试矩阵

### Smoke Test（烟雾测试）

所有开发完成后，必须执行 Smoke Test 来验证系统基本功能。

#### Smoke Test 执行规则

1. **必须手动验证**主要用户交互流程
2. **使用标准检查清单**（见项目文档）
3. **记录每个测试项的通过/失败状态**
4. **失败项必须追踪为 bug**

#### 标准检查清单项目

Smoke Test 应包含（由项目文档定义具体检查项）：

- ✅ 应用启动和基本功能
- ✅ 配置加载和数据持久化
- ✅ 核心业务流程
- ✅ 用户界面完整性
- ✅ 日志和错误处理
- ✅ 应用退出和清理

**项目特定的 Smoke Test 清单请参考项目文档** (通常在 `docs/README.md` 或对应的功能文档中)。

## 🧪 单元测试

### 编译 & 运行

```powershell
# 清理旧编译
dotnet clean

# 运行所有单元测试
dotnet test

# 运行特定项目的测试（示例）
dotnet test --project tests/[ProjectName].Tests.csproj

# 运行特定测试类（示例）
dotnet test --filter "TestClassName"

# 查看代码覆盖率
dotnet test /p:CollectCoverageRatio=true
```

### 通过标准

```
必须满足以下条件：

□ 所有测试通过（不能有失败）
□ 无新增测试失败
□ 代码覆盖率不下降
  - Models 层：≥ 90%
  - Service 层：≥ 80%
  - 总体：≥ 75%
```

## 🔍 集成测试

### 测试场景 1：完整代理生命周期

```
操作序列：
1. 添加代理（美国代理，8.8.8.8:8080）
2. 测试代理（验证连接）
3. 启动代理（状态变为 Running）
4. 检查本地端口（9001 应该被监听）
5. 停止代理（状态变为 Stopped）
6. 验证端口释放（9001 不再被监听）

预期结果：所有步骤成功
```

### 测试场景 2：多代理并发

```
操作序列：
1. 添加 3 个代理（不同的上游代理）
2. 全部启动
3. 验证 3 个本地端口都被监听（9001, 9002, 9003）
4. 添加第 4 个代理，测试端口分配
5. 全部停止
6. 验证端口都已释放

预期结果：端口分配正常，并发处理正确
```

### 测试场景 3：配置持久化

```
操作序列：
1. 添加代理 A、B、C
2. 关闭应用
3. 重新启动应用
4. 验证代理 A、B、C 仍然存在
5. 验证代理状态均为 Stopped（重启后状态复位）

预期结果：配置成功保存和加载
```

### 测试场景 4：错误恢复

```
操作序列：
1. 添加代理
2. 启动代理（观察日志，记录 PID）
3. 手动杀死 3proxy 进程
4. 等待监控服务检测（5-10 秒）
5. 验证代理状态变为 Failed

预期结果：故障检测正确
```

### 测试场景 5：日志完整性

```
在上述所有场景中：
1. 启用 DEBUG 日志
2. 执行完整操作
3. 查看 logs/ylproxy.log
4. 验证所有操作都有对应日志

期望的日志项：
- 应用启动
- 配置加载
- 代理添加
- 代理启动
- 3proxy 进程创建
- 端口监听验证
- 代理停止
- 应用退出
```

## 🚨 回归测试

### 修改代码后必须验证

```
修改前存在的功能 + 新修改的功能 = 完整功能

如果修改了 ProxyTester 性能优化：

□ 原有功能：代理测试仍然可用
□ 新增优化：耗时从 5s 降低到 3s
□ 无破坏：其他功能不受影响
```

## ✅ 测试检查表

### 每次开发完成时必须确认

```
□ 编译成功
  - dotnet build 返回 Exit Code = 0
  - 无编译错误
  - 无新增编译警告（必须修复或 @SuppressWarning）

□ 单元测试通过
  - dotnet test 返回 Exit Code = 0
  - 所有测试通过
  - 无新增失败

□ Smoke Test 通过
  - 应用正常启动
  - 主要功能可用
  - 日志无 ERROR

□ 功能完整性
  - 新增功能工作正确
  - 旧功能未破坏
  - 配置能保存和加载

□ 日志完整
  - 新功能有适当日志
  - 错误情况有异常日志
  - 没有过度日志污染
```

## 📊 测试失败处理

### 如果 Smoke Test 失败

```
步骤 1：重现失败
  - 按照测试步骤重现
  - 记录具体失败点

步骤 2：收集信息
  - 查看应用日志（logs/ylproxy.log）
  - 检查是否有异常
  - 记录错误信息

步骤 3：根本原因分析（RCA）
  - 是修改的代码导致？
  - 是环境问题（如端口占用）？
  - 是现有 bug？

步骤 4：修复
  - 如果是代码问题，修复代码后重新编译
  - 如果是环境问题，清理环境后重试
  - 回到第 1 步

步骤 5：验证
  - 重新运行失败的测试
  - 验证修复有效
```

## 📈 性能基准

### 性能指标（参考，非强制）

| 指标 | 目标 | 警告阈值 |
|---|---|---|
| **应用启动** | < 3 秒 | > 5 秒 |
| **配置加载** | < 500 ms | > 1 秒 |
| **代理测试** | < 10 秒 | > 15 秒 |
| **代理启动** | < 2 秒 | > 5 秒 |

如果超过警告阈值，建议进行性能分析。
