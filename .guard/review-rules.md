## 📋 完整自检清单

在标记任务为"完成"之前，**必须逐项确认以下清单**：

### 第 1 组：编译检查

```
□ 编译成功
  验证：dotnet build 返回 Exit Code = 0
  标准：无错误、无新增警告

□ 无编译错误
  验证：查看构建日志，确认"Build succeeded"
  标准：Error 计数 = 0

□ 无新增编译警告
  验证：构建日志中的 Warnings 部分
  标准：与基线相比无新增警告
  备注：如果需要抑制，使用 #pragma warning disable
```

### 第 2 组：测试检查

```
□ 单元测试全部通过
  验证：dotnet test 返回 Exit Code = 0
  标准：所有测试绿色（PASS）
  示例：25/25 tests passed

□ 无新增失败
  验证：与修改前的测试结果对比
  标准：失败数不增加

□ 代码覆盖率正常
  验证：运行覆盖率分析（如有工具）
  标准：不低于基线（Models ≥90%, Services ≥80%）
```

### 第 3 组：功能验证

```
□ Smoke Test 全部通过
  参考：.guard/test-rules.md 中的 Smoke Test 表
  标准：所有 12+ 个测试项都通过（✅）
  示例：
    ✅ 应用启动
    ✅ 配置加载
    ✅ 添加代理
    ...
    ✅ 应用退出

□ 新增功能工作正确
  验证：按需求规格执行新功能
  标准：功能行为符合预期

□ 旧功能未破坏
  验证：执行与修改相关的旧功能
  标准：旧功能仍然可用，无新增异常
```

### 第 4 组：代码质量

```
□ 编码规范已遵循
  检查项：
    - □ 命名规范（类/方法/字段）
    - □ 异常处理完整
    - □ 日志输出适当
    - □ 无硬编码配置
    - □ 无重复代码
  参考：.guard/coding-rules.md

□ 日志输出已验证
  验证：运行功能，查看 logs/ylproxy.log
  标准：
    - 有适当的 DEBUG 日志
    - 有操作成功的 INFO 日志
    - 有错误情况的 ERROR 日志
    - 没有过度日志污染

□ 异常处理完整
  检查项：
    - □ 没有空的 catch 块
    - □ 异常都有日志记录
    - □ 重要异常都已处理
```

### 第 5 组：文档更新

```
□ 代码注释已更新
  检查项：
    - □ 新方法有 XML 注释
    - □ 参数说明清晰（@param）
    - □ 返回值说明准确（@return）
    - □ 复杂逻辑有解释注释
  示例：
    /// <summary>
    /// 测试代理连接
    /// </summary>
    /// <param name="proxy">要测试的代理</param>
    /// <returns>测试结果，包含成功状态和耗时</returns>
    public async Task<ProxyTestResult> TestAsync(ProxyItem proxy)

□ 功能文档已更新
  如果修改或新增功能，对应的 .md 文件已更新
  示例：
    - 修改了 ProxyTester → 更新 03-架构设计/后端架构.md
    - 新增了 AutoRecovery → 创建对应功能文档

□ README 已更新
  如果涉及用户可见的功能变更
  示例：
    - 新增功能需要在 README 中说明
    - 配置变更需要更新配置说明

□ 更新日志已记录
  如果有 CHANGELOG.md，添加此次改动记录
```

### 第 6 组：修改范围

```
□ 修改文件清单完整
  检查项：
    - □ 列出所有修改的文件
    - □ 说明每个文件的修改理由
    - □ 没有无关修改
  示例：
    Modified Files:
    - ProxyTester.cs (性能优化)
    - AppSettings.json (新增 TestTimeout 配置)

□ 无无关修改
  检查项：
    - □ 没有修改其他功能的代码
    - □ 没有随意格式化代码
    - □ 没有意外修改
  提示：如果有无关修改，应该撤销（git checkout）
```

### 第 7 组：风险评估

```
□ 回归风险评估
  问题：
    - Q: 是否修改了公开 API？
    - Q: 是否改变了数据流？
    - Q: 是否影响了配置格式？
  结论：
    - 低风险：只修改了实现细节
    - 中风险：修改了内部接口
    - 高风险：修改了公开 API 或数据格式

□ 未发现回归问题
  验证：
    - □ 运行完整的 Smoke Test
    - □ 没有发现新的异常
    - □ 性能没有明显下降
    - □ 内存使用正常
```

### 第 8 组：最终审查

```
□ 生成的报告已审查
  参考：.guard/report-template.md
  标准：报告完整、准确

□ 所有检查都通过
  最终确认：
    - ✅ 编译：通过
    - ✅ 测试：通过
    - ✅ 功能：通过
    - ✅ 代码：通过
    - ✅ 文档：更新
    - ✅ 修改：清晰
    - ✅ 风险：可控
```

## ⚠️ 禁止标记完成的情况

### 如果以下任何一项为"❌"，**禁止**标记任务为完成

```
❌ 编译失败
❌ 测试失败
❌ Smoke Test 失败
❌ 代码不符合规范
❌ 文档未更新
❌ 发现回归问题
❌ 风险评估为高且未处理
```

**处理方法**：

1. 定位问题根本原因
2. 修复代码或文档
3. 重新运行验证
4. 回到对应步骤检查清单
5. 所有项都通过后再标记完成

## 🔄 自检流程

```
收到任务完成信号
    │
    ▼
执行第 1 组检查（编译）
    ├─ 全部通过？
    │  └─ 否 → 修复编译错误 → 重新检查
    │
    ▼
执行第 2 组检查（测试）
    ├─ 全部通过？
    │  └─ 否 → 修复失败测试 → 重新检查
    │
    ▼
执行第 3 组检查（功能）
    ├─ 全部通过？
    │  └─ 否 → 修复功能问题 → 重新检查
    │
    ▼
执行第 4 组检查（代码质量）
    ├─ 全部通过？
    │  └─ 否 → 调整代码 → 重新检查
    │
    ▼
执行第 5 组检查（文档）
    ├─ 全部通过？
    │  └─ 否 → 更新文档 → 重新检查
    │
    ▼
执行第 6 组检查（修改范围）
    ├─ 全部通过？
    │  └─ 否 → 调整修改 → 重新检查
    │
    ▼
执行第 7 组检查（风险）
    ├─ 全部通过？
    │  └─ 否 → 评估/处理风险 → 重新检查
    │
    ▼
执行第 8 组检查（最终）
    ├─ 全部通过？
    │  └─ 否 → 检查不通过 → 回到对应步骤
    │
    ▼
✅ 任务完成
```
