## 2. P0 阻塞性债务（7项，含新增）

### P0-1 🔴 API 层缺失 ProxyDataService 类 — **构建失败级**

**描述：** `ApiEndpoints.cs` 第 19 行引用 `ProxyDataService` 类，但该项目中 **不存在此类的任何定义**。同时 `YLproxy.Api.csproj` 缺少对 `YLproxy.Core` 和 `YLproxy.Proxy` 的 ProjectReference，导致编译直接失败。

**影响范围：** `src/YLproxy.Api/ApiEndpoints.cs:19`  
**风险等级：** 🔴 P0 — 构建阻断  
**预估工时：** 3h

**清偿方案：**
1. 在 `YLproxy.Core` 中创建 `ProxyDataService` 类（包装 `ProxyDataSerializer` + 原子文件读写 + 信号量锁）
2. 更新 `ApiEndpoints.cs` 中 `CreateSvc()` 的行内函数调用签名
3. 更新 `YLproxy.Api.csproj` 添加 ProjectReference 到 Core 和 Proxy

### P0-2 🔴 API 端口与代理端口范围重叠

**描述：** `AppSettings.json` 中 `Api.Port = 9100`，而 `Proxy.PortRangeEnd = 9100`（9001-9100）。当代理自动分配端口时可能分配 9100，导致 API Server 与代理进程端口冲突。

**影响范围：** `AppSettings.json:Api.Port`, `AppSettings.json:Proxy.PortRangeEnd`  
**风险等级：** 🔴 P0 — 运行时致命冲突  
**预估工时：** 0.5h

**清偿方案：**
1. 修改 `PortRangeEnd` 为 9099（或修改 `Api.Port` 为 9200）
2. 更新 `AppSettings.example.json` 保持一致
3. 更新 `PreFlightChecker.cs` 添加 API 端口与代理端口范围的非重叠校验

### P0-3 🔴 `_watcher.Changed` 事件重复触发导致配置抖动

**描述：** `FileSystemWatcher` 的 `Changed` 事件在保存时会触发多次（通常 2-3 次），且 `LoadConfig()` 每次执行都会读取文件。`SaveConfig()` 中文件写入后 FSW 会触发 `OnConfigChanged` → `LoadConfig()`。若 Save 过程中触发，可能捕获部分写入内容。

**影响范围：** `src/YLproxy.Infrastructure/AppSettingsService.cs:154-157`  
**风险等级：** 🔴 P0 — 运行时数据损坏风险  
**预估工时：** 1.5h

**清偿方案：**
1. 添加 `_lastLoadTime` 防抖（500ms 内不重复加载）
2. 原子写操作使用临时文件重命名（已有，但需要确保 FSW 不捕获 `.tmp` 文件）
3. 添加 `_isSaving` 标志跳过 FSW 事件

### P0-4 🔴 LoggerFactory 静态单例反 DI 模式 + 循环依赖风险

**描述：** `LoggerFactory.CreateLogger()` 内部读取 `AppSettings.json` 配置来初始化 `FileLogger`。如果被 `AppSettingsService` 构造函数调用（例如通过日志记录配置加载错误），会形成 **循环依赖**。

**影响范围：** `src/YLproxy.Infrastructure/LoggerFactory.cs:22`  
**风险等级：** 🔴 P0 — 死锁/栈溢出风险  
**预估工时：** 2h

**清偿方案：**
1. 将 LoggerFactory 改为 DI 注册的单例
2. 在所有项目中移除 `LoggerFactory.CreateLogger()` 静态调用
3. 使用更轻量的 NullLogger 作为回退，初始化完成后替换为 FileLogger

### P0-5 🔴 API 层 GlobalUsings 中使用 `ILog`/`LogFactory` 别名

**描述：** `GlobalUsings.cs` 定义 `ILog = YLproxy.Infrastructure.ILogger`（拼写为 ILog 而非 ILogger），在 `ApiServer.cs` 中使用了 `ILog` 类型。虽然在当前代码中能工作，但 `ILog` 别名在团队成员阅读时会产生混淆。

**影响范围：** `src/YLproxy.Api/GlobalUsings.cs:7`, `src/YLproxy.Api/ApiServer.cs`, `src/YLproxy.Api/ApiAuthMiddleware.cs`, `src/YLproxy.Api/ApiEndpoints.cs`  
**风险等级：** 🔴 P0 — 编译警告+维护混淆  
**预估工时：** 0.5h

**清偿方案：**
1. 将别名改为 `ILog = YLproxy.Infrastructure.ILogger` 可保留但文档消除
2. 或直接用 `YLproxy.Infrastructure.ILogger` 全限定名
3. 使用 `global using ILogger = YLproxy.Infrastructure.ILogger;`（与 Microsoft.Extensions.Logging.ILogger 不冲突，因为 API 项目仅引用自己的 Infrastructure）

### P0-6 🔴 `ProxyProcessManager` 全静态类不可测试

**描述：** `ProxyProcessManager` 所有方法均为 `static`，使用 `static` 的 `ConcurrentDictionary` 保存进程引用。这导致：
- 无法进行单元测试（Mock 进程行为不可能）
- 无法进行进程隔离
- DI 注入的 `IProxyProcessManager` 接口通过适配器委派给静态类，但测试时依然无法 Mock

**影响范围：** `src/YLproxy.Proxy/ProxyProcessManager.cs`  
**风险等级：** 🔴 P0 — 架构阻断  
**预估工时：** 4h

**清偿方案：**
1. 将 `Processes` 和 `Forwarders` 从 `static` 改为实例字段
2. 创建接口 `IProxyProcessManager` 的实例实现
3. 适配器模式保留静态 API 兼容性（已有 `ProxyProcessManagerAdapter`）
4. 将 `ProxyRuntimeConfiguration` 改为实例类以支持测试

### P0-7 🔴 `ApiEndpoints.cs` 中的重复 try-catch 模式

**描述：** 10 个端点中每个都有完全相同的 `try { ... } catch (Exception ex) { _logger.Error(...); return Results.InternalServerError(...); }` 模式，违反了 DRY 原则，且所有 catch 块都返回 `500 InternalServerError` 而非适当的 HTTP 状态码（如 503 或 502）。

**影响范围：** `src/YLproxy.Api/ApiEndpoints.cs:38-244` (10处)  
**风险等级：** 🔴 P0 — 代码膨胀+维护噩梦  
**预估工时：** 2h

**清偿方案：**
1. 创建 `ApiHelper.SafeEndpoint(Func<IResult>)` 包装方法
2. 使用 ASP.NET Core 的异常中间件
3. 或提取为扩展方法

## 3. P1 重要性债务（10项，含新增）

### P1-1 🟡 配置类位置错误

**描述：** `AppSettingsConfig`、`ProxyConfig`、`ThreeProxyConfig`、`LoggingConfig`、`StartupConfig`、`ApiConfig` 定义在 `Infrastructure` 层，但应定义在 `Models` 层（Models 层应只包含纯数据模型）。`Proxy.ProxyRuntimeConfiguration` 中的配置项与 `Infrastructure.ThreeProxyConfig` 重复。

**影响范围：**
- `src/YLproxy.Infrastructure/AppSettingsService.cs:199-239`
- `src/YLproxy.Proxy/ProxyRuntimeConfiguration.cs`

**预估工时：** 3h

**清偿方案：**
1. 将 6 个配置类移至 `src/YLproxy.Models/Config/` 目录
2. Infrastructure 项目引用 Models
3. 修复全仓 using/命名空间引用
4. `ProxyRuntimeConfiguration` 改为使用 `ThreeProxyConfig` 实例而非本地静态字段

### P1-2 🟡 `AppSettingsService.GetSection<T>()` 的非类型安全模式匹配

**描述：** `GetSection<T>` 使用 `switch` + `as T` 强制类型转换，若运行时类型不匹配静默返回 `new T()`（丢失数据）。违反类型安全原则。

**影响范围：** `src/YLproxy.Infrastructure/AppSettingsService.cs:56-66`  
**预估工时：** 1h

**清偿方案：**
1. 将配置类改为接口 + 具体实现
2. 或直接暴露强类型属性：`GetProxyConfig()` / `GetLoggingConfig()`
3. 使用 `JsonSerializer.Deserialize<T>()` 基于 section 名解析

### P1-3 🟡 `MonitorService.CheckHealth` 重用 `_restartBackoff` 作为健康检查时间戳

**描述：** `CheckHealth` 方法（第 150 行）使用 `_restartBackoff` 的 `LastAttempt` 字段作为"最近健康检查时间"的存储。这会与 `TryAutoRestart` 方法的退避时间戳冲突：如果健康检查先写，退避判断会错误使用健康检查时间而非重启尝试时间。

**影响范围：** `src/YLproxy.Core/MonitorService.cs:150-154`  
**预估工时：** 1.5h

**清偿方案：**
1. 添加独立的 `ConcurrentDictionary<int, DateTime>` 作为健康检查时间戳存储
2. 将健康检查时间与重启退避时间分离

### P1-4 🟡 `ManagedProxyForwarder` 中的 `_logger` 静态引用

**描述：** `HandleClientAsync` 中第 182 行 `_logger.Debug($"Failed to add header ...")`，以及在 `WriteError` 中第 306 行调用 `_logger` 静态方法，但 `WriteError` 是 `static` 方法，无法访问实例 logger。这导致日志记录行为不一致。

**影响范围：** `src/YLproxy.Proxy/ManagedProxyForwarder.cs:300-307`  
**预估工时：** 0.5h

**清偿方案：**
1. 将 `WriteError` 改为非静态方法（或其 logger 参数改为从 `ILogger` 实例传入）
2. 统一日志前缀格式

### P1-5 🟡 `MonitorService` 中 `_refreshAction` 在 `MonitorTick` 中的调用路径过深

**描述：** `MonitorTick` → `CheckHealth` → `TryAutoRestart` 三层都有可能调用 `_refreshAction()` 和 `_saveAction?.Invoke()`。这可能导致 WPF UI 线程过度刷新（5 秒间隔内多次调用 DataGrid 刷新）。

**影响范围：** `src/YLproxy.Core/MonitorService.cs`  
**预估工时：** 1h

**清偿方案：**
1. 使用 `MonitorTick` 级别的单个 `changed` 标志，在 tick 结束后统一刷一次
2. `CheckHealth` 和 `TryAutoRestart` 仅设置 `changed=true`

### P1-6 🟡 API 层 Swagger 默认启用但无生产环境开关

**描述：** `ApiServer.cs` 中 Swagger 默认禁用（`enableSwagger = false`），但 API 路由中 `/swagger` 在 `ApiAuthMiddleware` 中被豁免鉴权。若生产环境错误启用 Swagger，将暴露完整的 API 文档和 schema。

**影响范围：** `src/YLproxy.Api/ApiServer.cs:54`, `src/YLproxy.Api/ApiAuthMiddleware.cs:24`  
**预估工时：** 0.5h

**清偿方案：**
1. 在 `ApiAuthMiddleware` 中移除 swagger 豁免或默认禁用
2. 添加环境配置开关（`ASPNETCORE_ENVIRONMENT=Production` 时禁用 Swagger）
3. 将 `enableSwagger` 参数改为从 `AppSettings.json` 读取

### P1-7 🟡 配置验证中的硬编码字符串

**描述：** `AppSettingsService.Validate()` 中大量硬编码 Magic String：`"logs"`, `"data"`, `"config.json"`, `"runtime/3proxy"` 等。违反 DRY 原则，且与 `AppSettingsConfig` 的默认值重复。

**影响范围：** `src/YLproxy.Infrastructure/AppSettingsService.cs:168-195` (6处)  
**预估工时：** 1h

**清偿方案：**
1. 定义 `public const string` 常量类（如 `ConfigDefaults`）
2. 验证方法引用常量而非直接使用 magic string

### P1-8 🟡 `ProxyTester.TestOnceAsync` 中的空异常捕获

**描述：** 第 191-192 行 `catch (Exception) { return (false, 0, "连接失败"); }` 丢弃了异常详细信息。这是宽泛的空 catch，掩盖了真实错误原因。

**影响范围：** `src/YLproxy.Core/ProxyTester.cs:190-193`  
**预估工时：** 0.5h

**清偿方案：**
1. 捕获具体异常类型
2. 至少记录异常到日志
3. 使用 `ex.Message` 作为 error 消息

### P1-9 🟡 缺少配置变更的审计日志

**描述：** API 层对代理的 CRUD 操作（添加、删除、启动、停止）虽然记录 Info 日志，但未记录操作来源（IP、User-Agent）和执行详情。AppSettingsService 的热加载变更也缺少审计（谁在何时修改了文件）。

**影响范围：** 全仓 API 和配置变更路径  
**预估工时：** 2h

**清偿方案：**
1. 在 `ApiEndpoints` 中添加请求上下文（来源 IP、User-Agent）日志
2. 在 `AppSettingsService.OnConfigChanged` 中添加文件变更审计日志
3. 在 ProxyItem 上添加 `LastModifiedTime` 和 `LastModifiedBy`

### P1-10 🟡 测试覆盖率缺口（新增 5 个关键缺口）

## 4. P2 优化性债务（12项，含新增）

### P2-1 ️🟢 `SimpleRetry` 使用 `Thread.Sleep` 阻塞线程

**描述：** `SimpleRetry.Execute` 同步重试方法使用 `Thread.Sleep` 阻塞当前线程。在 WPF UI 线程上调用会阻塞 UI。

**影响范围：** `src/YLproxy.Infrastructure/SimpleRetry.cs:41`  
**预估工时：** 1h

### P2-2 ️🟢 文件操作缺少异步版本

**描述：** `ProxyDataSerializer` 的 `Serialize`/`Deserialize` 全部同步。对于可预见的未来 JSON 文件（~100 条代理），同步操作在当前规模没问题，但架构上不完整。

**影响范围：** `src/YLproxy.Core/Config/ProxyDataSerializer.cs`  
**预估工时：** 1h

### P2-3 ️🟢 `ProxyProcessManager.IsRunning()` 中 `Ensure3ProxyDependencies()` 副作用

**描述：** `IsRunning()` 作为只读查询方法，却调用了 `Ensure3ProxyDependencies()` 进行文件系统操作（检查 exe 和 DLL 是否存在），这产生了副作用。每次 `MonitorTick` 都会检查，产生不必要的 I/O。

**影响范围：** `src/YLproxy.Proxy/ProxyProcessManager.cs:320-332`  
**预估工时：** 1h

### P2-4 ️🟢 `ConfigGenerator` 中 3proxy 日志路径硬编码

**描述：** `GetRuntime3ProxyPath()` + `logs\\3proxy-{proxy.Id}.log` 路径基于运行时目录而非配置的日志目录。日志可能被规则删除/轮转遗漏。

**影响范围：** `src/YLproxy.Proxy/ConfigGenerator.cs:31`  
**预估工时：** 0.5h

### P2-5 ️🟢 `ManagedProxyForwarder` 请求体缓冲区硬限制 64KB

**描述：** `reqBuf = new byte[65536]` 限制了请求体大小。对于大 POST，即使有 body 也会截断。

**影响范围：** `src/YLproxy.Proxy/ManagedProxyForwarder.cs:115`  
**预估工时：** 1.5h

**清偿方案：**
1. 改用流式读取头部（`StreamReader.ReadLineAsync`）
2. body 使用 `CopyToAsync` 分块转发

### P2-6 ️🟢 `ExceptionHandler` 中 `OnUserNotification` 空 catch

**描述：** `ExceptionHandler.Handle` 中第 26 行 `catch {}` 尝试调用 `OnUserNotification` 失败时静默丢弃。虽然意图是"不掩盖原始异常"，但意外丢失了通知失败信息。

**影响范围：** `src/YLproxy.Infrastructure/ExceptionHandler.cs:26-28`  
**预估工时：** 0.5h

### P2-7 ️🟢 项目间耦合方向违反 DDD 原则

**描述：** `YLproxy.Core` 依赖 `YLproxy.Infrastructure` 和 `YLproxy.Proxy`，但 `Infrastructure` 应该是最底层，Core 依赖 Infrastructure 是合理的，但 Core 依赖 Proxy 表明 Core 层需要提取进程管理相关的抽象。

**影响范围：** `src/YLproxy.Core/YLproxy.Core.csproj`（依赖 Infrastructure/Proxy/Utils/Models）  
**预估工时：** 3h

### P2-8 ️🟢 缺少 `data/README.md` 中提到的数据文件说明

**描述：** 文档中提到数据目录的描述，但实际 `data/README.md` 可能未与代码同步。

## 5. P3 风险性债务（8项，新增）

### P3-1 🔵 `ManagedProxyForwarder` 中 `Task.Run` 无限制并发

**描述：** `AcceptLoopAsync` 中对每个客户端连接使用 `Task.Run(() => HandleClientAsync(...))`，但完全没有并发限制（如 `SemaphoreSlim` 或 `ParallelOptions.MaxDegreeOfParallelism`）。高并发时可能导致线程池饥饿和 OOM。

**影响范围：** `src/YLproxy.Proxy/ManagedProxyForwarder.cs:95-97`  
**预估工时：** 2h

### P3-2 🔵 CONNECT 隧道响应解析使用 Contains("200") 而非精确解析

**描述：** `HandleConnectAsync` 中第 247 行 `respText.Contains("407")` 和第 264 行 `respText.Contains("200")` 使用字符串包含判断，可能误匹配（如响应体包含"200"）。同样的问题在重试判断中也存在。

**影响范围：** `src/YLproxy.Proxy/ManagedProxyForwarder.cs:247,264`  
**预估工时：** 1h

**清偿方案：**
1. 使用正则或精确解析 HTTP 状态码行
2. 解析 `HTTP/1.1 200` 而非子串包含

### P3-3 🔵 `ProxyProcessManager` 的 `Process.Kill(true)` 可能导致子进程僵尸

**描述：** `Kill(true)` 仅杀进程树。但 3proxy 可能启动额外的工作进程（`service` 模式），即使 `Kill(true)` 也无法保证完全清理。没有进程退出等待确认。

**影响范围：** `src/YLproxy.Proxy/ProxyProcessManager.cs:384-396`  
**预估工时：** 1.5h

### P3-4 🔵 `AppSettingsService.Validate` 中 `Logging.LogDirectory` 必须等于 "logs" 的强约束

**描述：** 验证方法强制 `Logging.LogDirectory` 必须为 "logs"，不允许用户自定义日志路径。这在容器化部署中是不必要的限制。

**影响范围：** `src/YLproxy.Infrastructure/AppSettingsService.cs:168-170`  
**预估工时：** 0.5h

### P3-5 🔵 `FileLogger` 使用 `lock(_lock)` 而非 `ReaderWriterLockSlim`

**描述：** 所有日志写入使用 `lock` 语句控制并发，且写操作前后还包含外部 `try-catch` 结构。高并发日志场景下锁争用可能成为瓶颈。

**影响范围：** `src/YLproxy.Infrastructure/FileLogger.cs:17,126-140`  
**预估工时：** 1.5h

### P3-6 🔵 `DpapiSecurityService` 中 `Encrypt` 方法对已加密字符串的"透明"重加密保护

**描述：** `Encrypt` 第 18-22 行检查输入是否已是加密格式，如果是则直接返回（不做二次加密）。但如果输入的是无效加密字符串（如以 `dpapi:v1:` 开头但 payload 损坏），会静默返回无效加密字符串而非抛出异常。`Decrypt` 方法会正常抛出异常，但这产生了不一致的防御行为。

**影响范围：** `src/YLproxy.Infrastructure/DpapiSecurityService.cs:18-23`  
**预估工时：** 0.5h

### P3-7 🔵 `ProxyProcessManager` 中 `Ensure3ProxyDependencies` 在非 Windows 平台会失败

**描述：** 检查 3proxy.exe 文件，但在 Linux/macOS 容器中没有 exe 文件。对于 Docker 部署场景需要条件编译或平台判断。

**影响范围：** `src/YLproxy.Proxy/ProxyProcessManager.cs:54-60`  
**预估工时：** 2h

### P3-8 🔵 `ManagedProxyForwarder.STOP` 中 `_cts.Cancel()` 未与 `AcceptLoopAsync` 的 `while(!token.IsCancellationRequested)` 协同

**描述：** `Stop()` 中 `_cts.Cancel()` 后调用 `_listener.Stop()`，但 `AcceptLoopAsync` 的 `continue` 分支在 `catch{}` 中无 `break`，可能导致空循环继续运行。

**影响范围：** `src/YLproxy.Proxy/ManagedProxyForwarder.cs:310-323`  
**预估工时：** 1h
