## 一、工程债分析

### 1.1 基础设施债

#### 1.1.1 数据持久化策略未决 🔴 P0

**问题描述:**
- 当前使用 JSON 文件存储 (`data/config.json`)
- SQLite 迁移代码已存在但未启用 (`SqliteProxyRepository`, `DataMigrationService`)
- `ProxyDataService.MigrateToSqliteIfNeeded()` 返回 `false`，迁移逻辑被跳过
- 存在双写代码分支但未实际执行

**影响:**
- 缺少事务支持，数据一致性风险
- 并发访问冲突风险 (虽有 `_ioLock` 保护)
- 无查询能力，无法支持复杂筛选和统计
- 配置格式变更无版本化迁移机制

**债务规模:** 大 (涉及核心数据层，影响全局)

**建议方案:**
- **方案 A (推荐):** 废弃 SQLite 迁移，优化 JSON 持久化
  - 理由: 当前单机应用场景 JSON 足够，SQLite 引入维护负担
  - 行动: 删除 SQLite 相关代码，简化 `ProxyDataService`
  - 工时: 4h
  
- **方案 B (积极):** 完成 SQLite 切换
  - 理由: 为未来多实例共享和复杂查询做准备
  - 行动: 启用迁移逻辑，JSON 降级为备份
  - 工时: 8h

#### 1.1.2 依赖版本管理 🟡 P1

**问题描述:**
- `.NET 10.0.200` 在 `global.json` 中固定，但 `Directory.Packages.props` 引用 `Microsoft.Extensions.DependencyInjection` 9.0.0
- SQLite 相关包版本 2.1.12 (修复 NU1903 漏洞)
- 缺少依赖更新自动化机制

**影响:**
- 潜在的兼容性风险
- 安全漏洞响应滞后

**债务规模:** 小

**建议方案:**
- 建立 Dependabot 或 Renovate 配置
- 定期审查 NuGet 包安全公告
- 工时: 2h

#### 1.1.3 CI/CD 发布自动化缺失 🟡 P1

**问题描述:**
- 现有 CI 仅完成构建和测试
- 无发布打包步骤
- 无版本号自动管理
- 无 SBOM 生成
- 无代码签名

**影响:**
- 发布过程手动化，容易出错
- 缺少软件物料清单 (SBOM)
- 用户无法验证发布完整性

**债务规模:** 中

**建议方案:**
- 新增 `release.yml` workflow
- 集成 `dotnet publish` 单文件打包
- 自动生成 GitHub Release
- 集成 CycloneDX 生成 SBOM
- 工时: 6h

### 1.2 配置管理债

#### 1.2.1 配置类位置不当 🟢 P2

**问题描述:**
- `AppSettingsConfig`, `LoggingConfig`, `ProxyConfig`, `ThreeProxyConfig` 等配置类定义在 `YLproxy.Infrastructure` 项目中
- 违反分层原则，配置类应放在 `YLproxy.Models` 中

**影响:**
- 依赖方向不清晰
- Models 项目无法保持零依赖

**债务规模:** 小

**建议方案:**
- 将配置类迁移至 `src/YLproxy.Models/Config/`
- 更新所有 using 引用
- 工时: 2h

#### 1.2.2 FileSystemWatcher 线程安全 🟡 P1

**问题描述:**
- `AppSettingsService` 使用 `FileSystemWatcher` 监听配置变更
- 配置加载缺少线程同步机制
- 可能存在竞态条件

**影响:**
- 配置热加载时可能出现数据不一致
- 低概率但影响严重

**债务规模:** 中

**建议方案:**
- 引入 `ReaderWriterLockSlim` 保护配置读写
- 热加载时获取写锁
- 读取配置时获取读锁
- 工时: 2h

### 1.3 安全债

#### 1.3.1 空 catch 块隐藏异常 🔴 P0

**问题描述:**
- 全项目存在 13 处空 catch 块
- 分布: `MainViewModel.cs` (4处), `ManagedProxyForwarder.cs` (4处), `TransparentCoalescingForwarder.cs` (3处), `AesSecurityService.cs` (1处), `AppSettingsService.cs` (1处)
- 部分场景合理 (进程清理)，但多数应记录日志

**影响:**
- 异常被静默吞没，排查困难
- 可能隐藏严重错误

**债务规模:** 中

**建议方案:**
- 逐个审查空 catch 块
- 合理场景加注释说明
- 其他场景至少记录 Warn 级别日志
- 工时: 3h

#### 1.3.2 DPAPI 跨用户迁移限制 🟢 P2

**问题描述:**
- 凭据使用 Windows DPAPI `CurrentUser` 加密
- 无法跨用户或跨机器迁移配置
- 需要重新录入凭据

**影响:**
- 用户体验受限
- 但符合安全设计意图

**债务规模:** 小 (设计限制，非缺陷)

**建议方案:**
- 在文档中明确说明限制
- 提供凭据导出/导入工具 (可选)
- 工时: 4h (可选)
